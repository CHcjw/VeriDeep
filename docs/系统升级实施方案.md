# VeriDeep 多 Agent 竞品调研系统 · 全面升级实施方案

> 本文档记录VeriDeep 从「能端到端跑通」到「可观测、可回放、带返工闭环的 Deep Research 系统」的完整设计与演进过程，是 AI 协作（TRAE）深度参与下的架构决策与实施记录。

## Context（背景与目标）

这是一套多 Agent 竞品调研系统（FastAPI 后端 + React/Vite 前端 + 智谱 GLM）。早期版本已能端到端跑通，但存在阻塞性问题与大量能力缺口：

**阻塞性问题（必须先修）**
1. **报告章节空洞**：9 章在一次 `max_tokens=8000` 的串行 LLM 调用里写，思考模式吃光 token，章节掉到一句话兜底。
2. **写报告太慢**：9 章串行 + 单并发模型 → 十余分钟未完成。目标压缩到 3–5 分钟。
3. **搜索质量**：题不对版、网页打不开、抓取乱码、数据陈旧。
4. **可信度造假**：写死 0.85/0.5/0.7，全是 5 的倍数，看着假。

**能力缺口（19 项）**：可观测性 Trace、决策回放、结构化消息/Schema、真实反馈闭环、量化提升指标、业务闭环指标、数据空间 CSV、调研模式分档、批注驱动二次调研、创新报告板块等。

**已确认的关键决策**
- 执行节奏：一次性全做完（但内部按依赖顺序分阶段）。
- 模型策略：核心章 GLM-5.2 + 辅助章 GLM-5.1 + 杂务用 GLM-Z1-Air（5.2/5.1 各 10 并发，Z1-Air 30 并发，多章全并行无压力）。
- 调研模式：快速/深度/专家级三档，按「搜索量 + 章节数 + 模型档位」分档。
- 可观测性：报告页决策回放 + 工作台悬浮可拖拽日志面板 + 独立 Trace 页签。

**关键技术前提**：`run_pipeline` 所有同步调用用 `asyncio.to_thread` 包裹，Python 3.9 的 `to_thread` 会复制 contextvars 进线程，因此可用 contextvar 做**无侵入埋点**（无需给每个 `chat()` 改签名传 agent_id）。报告整份 JSON 存 `reports.data` 并随 `GET /api/reports/:id` 下发，所以挂在报告里的新结构（trace/metrics/structured/data_grid）前端零额外请求即可用。

| 模型 | 并发 | 用途 |

## 核心架构改造

### 阶段 0 · 底座与基础修复（一切的前置）

**0.1 `llm.py` 支持多模型 + Trace 埋点**（[backend/app/core/llm.py](../backend/app/core/llm.py)）
- `chat / chat_json / chat_stream` 增加 `model: Optional[str] = None`，`use_model = model or settings.zhipu_model`。
- `chat()` 内包计时，成功后除 `TOKEN_USAGE` 累加外，调 `trace.record_span(...)`。一处埋点全覆盖。
- 修正 thinking 模式判断：带思考的模型 `max_tokens` 要足够大（输出占比较低，需 ≥6000 才有实际产出）。

**0.2 新建 [trace.py](../backend/app/core/trace.py)**
- `TraceSpan` dataclass：`span_id, task_id, seq, agent_id, stage, purpose, model, prompt(摘要), response(摘要), prompt_tokens, completion_tokens, total_tokens, latency_ms, decision, evidence_ids, ts`。
- contextvar `_CTX` + `_BUFFER: Dict[task_id, List]` + Lock。
- `set_context()` / `record_span()` / `drain(task_id)`（增量）/ `get_trace(task_id)`（全量）。

**0.3 可信度真实计算 — 新建 [credibility.py](../backend/app/core/credibility.py)**
- `score_evidence(...) -> int`（0–100，精确到个位）：
  - 分级基分：official 70 / 财报 75 / news 60 / zhihu 50 / bilibili 45 / weibo 40 / xiaohongshu 38 / douyin 35 / review·unknown 30。
  - 域名权威性：gov.cn/.edu/官网/头部财经媒体 +10；聚合站/营销号 -5。
  - 发布时间 +6；时效性：1 年内 +5、1–2 年 +2、>3 年 -5。
  - 抓取质量：正文成功 +6；仅 snippet -8；excerpt>200 字 +3。
  - `max(5, min(98, score))`。
- `Evidence.credibility` 语义为 0–100 int，新增 `freshness_days`。前端阈值从 0–1 改 0–100。

**0.4 乱码/相关性过滤 — 新建 [textquality.py](../backend/app/core/textquality.py)**
- `is_garbled(text)`：替换符占比/不可打印字符/CJK 比例阈值。
- `is_relevant_content(text, brands, query)`：抓回正文二次校验品牌/关键词命中。
- 在 `fetcher._extract_text` 加 `apparent_encoding` 兜底解码 + 乱码丢弃。

**0.5 数据时效性**（[search.py](../backend/app/core/search.py)）
- `freshness` 按调研模式传入（深度/专家用 `"oneYear"`，部分「最新动态」角度用 `"oneMonth"`）。
- 搜索角度注入年份关键词提升新鲜度。

### 阶段 4 · 量化提升 + 业务闭环指标

**新建 [metrics.py](../backend/app/core/metrics.py)**
- 真实数据源：elapsed_seconds、token 增量、独立信源数、平台数、高置信比、返工轮数、人工修正数。
- 公式（透明、基线可配置并标注）：
  - 效率提升 = 人工分钟 / 实际分钟
  - 覆盖度 = 独立域名数 / 人工基线源数
  - 一致性 = 0.5×(挂证据 claim 比) + 0.5×(schema 字段填充率)
  - 准确率 = high_conf/total；人工修正率 = edited_blocks/total_blocks
- `POST /api/reports/:id/feedback` + `report_feedback` 表。
- 前端 `VMetricsPanel.tsx` + DashboardPage 聚合，每项 tooltip 展示输入（可解释）。

## 验证方式（端到端）

1. **基础修复验证**：重启 `./restart.sh`，跑深度模式，确认：写报告 ≤90 秒；每章 ≥4 段非空；可信度分数有差异；无乱码证据；搜索结果相关。
2. **时效性**：检查证据 captured_at / 标题含近期新内容占比。
3. **可观测性**：工作台悬浮面板实时滚动 trace；报告页决策回放可步进高亮证据；`/trace/:reportId` 可查每个 Agent 的 prompt/输出/token。
4. **结构化**：feature/pricing/persona 章渲染功能树/定价表/画像卡；data_grid 可展开导出 CSV。
5. **闭环**：质检触发返工，前后指标 delta 可见。
6. **量化指标**：报告页 metrics 面板显示各项指标，tooltip 可解释。
7. **三档模式**：快速 ~2 分钟 5 章；专家级 ~6–8 分钟 11+ 章含创新板块。
8. **批注**：划线持久高亮 + 内联批注 + 「按批注深化」触发二次调研更新章节。
9. **稳定性**：连续跑 2–3 次不崩、不卡死。

## 风险与权衡

- **专家级 6–8 分钟接近红线**：演示建议用深度模式（~4 分钟）兜底，专家级作为能力展示备选。
- **并行多章 + 补采 + Trace**：需确认账户并发实测，span 用 `seq` 全局递增保证回放顺序正确。
- **工作量大**：阶段 0→6 是「能演示」的最小完整闭环，7→10 是加分项。
